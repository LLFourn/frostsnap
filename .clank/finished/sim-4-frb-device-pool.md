# sim-4-frb-device-pool
# sim-4: Slice 2a — FRB `DevicePool`/`VirtualDevice` + debug `load_sim()` entrypoint

## Goal
Bind the **real** app coordinator to one virtual device through flutter_rust_bridge. Add a
debug-only entrypoint that hands Dart two independent objects — the unchanged `Coordinator` and
a `DevicePool` — and the FRB opaque types the tray (sim-5) and driver (sim-7) build on.

## What sim-1/2/3 already built (don't redo)
`VirtualDevice` (real `DeviceLoop`+`FrostyUi`), `session()`/`poll_once`, `framebuffer()`
(`export_rgba`/`save_png`/`take_dirty`), `touch()`, `host_serial() -> HostEnd`; `VirtualSerial`
(`Serial`) / `VirtualPort` transport with the Condvar blocking read; `SimFirmware` (upgrades
never offered). NOTE: sim-3's `SimCoordinator` is the *pure-Rust test* driver — the **app uses
the real `FfiCoordinator`**, so sim-4 does **not** use `SimCoordinator`; it seeds
`pool.serial()` into the app's real `UsbSerialManager`.

## Core model (read first)
**Two surfaces, reconciled by `DeviceId`, no central router (Axis 4).** The app gets the normal
`Coordinator` (built exactly as `load()` does, `frostsnapp/rust/src/api/init.rs`) plus a
`DevicePool` that owns the per-thread virtual device(s) and exposes `serial()` — the
`VirtualSerial` seeded into `UsbSerialManager`. The two never talk directly: the `VirtualDevice`
generates the device keypair; the coordinator learns the same id via `Announce`. **Plug/unplug is
just the `available_ports()` set** — no faked `DeviceChange`, no coordinator edits. **One device,
always connected** (pool size 1). Each device is its own `#[frb(opaque)]` handle (mirror
`FfiSerial`: `frostsnapp/rust/src/api/port.rs:25-107` — `Arc<Mutex<…>>`, `StreamSink`).

## The device must run on its OWN thread, built IN that thread (FrostyUi is !Send)
The app's `FfiCoordinator` runs its poll loop on a **background thread**
(`frostsnapp/rust/src/coordinator.rs`), so the virtual device must run concurrently on its **own
thread** (sim-2/3's single-threaded lockstep pump does not apply to the live app). But `FrostyUi`
is `!Send` (`Rc`), so a `VirtualDevice` **cannot be moved** into `thread::spawn`. The committed
shape (deferred here from sim-2): a device-thread runtime in `tools/virtual_device` that
**constructs the `VirtualDevice` *inside* the spawned thread** from a seed, then hands the
`Arc`-backed handles (`framebuffer()`, `touch()`, `host_serial()` — all `Send`) back out via a
channel, then runs `loop { poll_once(false); on take_dirty → push frame; park briefly }`. Nothing
`!Send` ever crosses the thread boundary; the device and the coordinator thread communicate only
through the `Arc`-backed pipe (the Condvar blocking read from sim-2). A stop flag + join on drop.
(ruthless sim-2 note: replace the idle busy-loop with a short park — done here.)

## FRB codegen (the build mechanics)
`frostsnapp/rust/src/frb_generated.rs` is a **gitignored generated artifact** — it's absent in a
fresh worktree, which is why `cargo check` at the workspace root currently fails there. It is
regenerated by **`just gen`** (`flutter_rust_bridge_codegen generate`, tooling confirmed present:
`flutter`/`dart`/`flutter_rust_bridge_codegen`). So: add the Rust api, run `just gen`, then
`cargo check -p rust_lib_frostsnapp` builds. **Reviewers must `just gen` before checking the app
rust crate** (the generated file isn't committed). The generated Dart bindings land under
`frostsnapp/lib/src/rust/` (used by sim-5).

## Tasks
1. `frostsnapp/rust` depends on `frostsnap_virtual_device` (SDL-free per sim-1 — the app build
   stays SDL-free; verify with `cargo tree`).
2. Device-thread runtime in `tools/virtual_device` (the build-in-thread pattern above): a
   `spawn(seed) -> (DeviceHandle, framebuffer, touch, host_serial)` (or similar) that builds the
   device in-thread, returns the `Send` handles, runs the poll loop, and stops/join on drop.
3. `#[frb(opaque)] VirtualDevice { id() -> DeviceId; frames() -> StreamSink<Frame>;
   touch(x, y, lift_up); }` where `Frame` carries RGBA bytes + dims (`export_rgba`); push frames
   on `take_dirty()` from the device thread into the registered `StreamSink`. (`plug`/`unplug`
   deferred — size-1, always connected.)
4. `#[frb(opaque)] DevicePool` owning the device handle(s) + the `VirtualSerial`; `serial()`
   yields the seed for `UsbSerialManager`; `devices() -> Vec<VirtualDevice>` for the tray.
5. `load_sim()` debug entrypoint in `api/init.rs` beside `load()`: build the coordinator via the
   existing `load_internal` but pass `Box::new(pool.serial())` to `UsbSerialManager::new` instead
   of `DesktopSerial` (and **no firmware bin** → the manager never offers an upgrade, consistent
   with the sim guard). Return `(Coordinator, AppCtx, DevicePool)`. Gate so production can't reach
   it.
6. Run `just gen`.

## Acceptance
- `just gen` succeeds; `cargo check -p rust_lib_frostsnapp` builds (with the generated
  `frb_generated.rs`); the app build pulls **no** SDL (`cargo tree`).
- A Rust test: a `DevicePool` of size 1, its device on its own thread, registers through a real
  `UsbSerialManager` (`DeviceChange::Registered`) via these FFI types, and a frame is observable
  (`export_rgba`/the frames path yields ≥1 RGBA frame).
- `load()` / `load_host_handles_serial()` and all production paths are untouched; esp still builds.

## Non-goals / deferred
No Dart UI yet (sim-5). No multi-device pool / dynamic `add_device`/`unplug` (size 1, connected).
No signing. The `SimCoordinator`→`frostsnap_coordinator` convergence (sim-3 note) stays deferred —
the app uses the real `FfiCoordinator`.
